# System Overview: Process Manager

## Purpose

The `Process_Manager` is the central hub for all cooperative task scheduling
in the engine. It owns the pool of all `Process` instances, the
`Process_Table` that schedules them, and the UUID generation state. It
provides the public API for allocating, running, polling, and releasing
processes.

The `Process_Manager` lives in the `Game` struct and is the only process
system component that game code and engine subsystems interact with
directly.

## Architecture

### Data Layout

    Game
    +-- Process_Manager process_manager
        +-- Process processes[PROCESS_MAX_QUANTITY_OF]
        |   +-- [0] Process (slot 0)
        |   +-- [1] Process (slot 1)
        |   +-- ...
        |   +-- [511] Process (slot 511)
        |
        +-- Process_Table process_table
        |   +-- Process_Priority_Table_Entry[PROCESS_MAX_PRIORITY_LEVEL + 1]
        |   +-- Process *ptr_array_of__processes[PROCESS_MAX_QUANTITY_OF]
        |   +-- Process_Priority_Table_Entry *p_process_priority_table_entry__current
        |
        +-- Repeatable_Psuedo_Random repeatable_psuedo_random_for__process_uuid
        |   (deterministic UUID generator)
        |
        +-- Process *p_process__latest
        |   (one-shot pointer to most recently allocated process)
        |
        +-- Identifier__u32 next__uuid__u32
            (next UUID seed)

### Key Types

| Type | Role |
|------|------|
| `Process_Manager` | Top-level owner. Holds the process pool, the scheduling table, and UUID state. |
| `Process` | Cooperative task unit. See `system_overview__process_unit.md`. |
| `Process_Table` | Priority-partitioned scheduling structure. See `system_overview__process_table.md`. |
| `Repeatable_Psuedo_Random` | Deterministic pseudo-random number generator for UUID generation. |

### Limits

| Macro | Default | Description |
|-------|---------|-------------|
| `PROCESS_MAX_QUANTITY_OF` | 512 | Size of the process pool. |

## Pool Management

### Allocation Strategy

The process pool is a contiguous array of `Process` structs. Each slot is
either available (`Process_Status_Kind__None`, UUID = `IDENTIFIER__UNKNOWN__u32`)
or occupied (any other status, UUID assigned).

Allocation uses hashed lookup via `dehash_identitier_u32_in__contigious_array`
to find a slot matching the requested UUID. If the UUID collides with an
occupied slot, the hash function probes for an available slot.

### UUID Generation

UUIDs are generated by the `Repeatable_Psuedo_Random` generator, ensuring
deterministic allocation order for reproducible behavior. The
`next__uuid__u32` field tracks the seed state.

### Latest Process Pattern

After allocation, `p_process__latest` points to the newly allocated process.
This enables a one-shot consumption pattern:

    run_process(&pm, handler, priority, flags);
    Process *p = get_p_latest_allocated_process_from__process_manager(&pm);
    // p is the process just allocated.
    // p_process__latest is now NULL.
    // Calling again returns NULL until the next allocation.

This pattern avoids the need to return the process pointer through complex
call chains (e.g. when allocation happens deep in a subsystem).

## Lifecycle

### 1. Initialization

    initialize_process_manager(&game.process_manager);
        -> All 512 process slots: initialized as empty
           (UUID = IDENTIFIER__UNKNOWN__u32, status = None).
        -> Process_Table: initialized (all priority entries set up,
           pointer array cleared).
        -> Repeatable_Psuedo_Random: seeded.
        -> p_process__latest = NULL.
        -> next__uuid__u32 = initial seed.

### 2. Process Allocation (Low-Level)

    Process *p_proc = allocate_process_in__process_manager(
            &process_manager,
            uuid,
            priority_level);
        -> Finds an available slot via hashed lookup.
        -> Sets the slot's UUID and priority.
        -> Does NOT set the run handler or add to the process table.
        -> Sets p_process__latest.
        -> Returns NULL if the pool is full.

This is the low-level allocator. Most callers should use `run_process` or
`run_process_with__uuid` instead.

### 3. Running a Process (High-Level)

    // Auto-generated UUID:
    Process *p_proc = run_process(
            &process_manager,
            my_handler,
            PROCESS_PRIORITY__0_MAXIMUM,
            PROCESS_FLAGS__NONE);

    // Specific UUID:
    Process *p_proc = run_process_with__uuid(
            &process_manager,
            my_handler,
            my_uuid,
            PROCESS_PRIORITY__1,
            PROCESS_FLAG__IS_CRITICAL);

Both functions:
1. Allocate a process slot (generating a UUID if needed).
2. Initialize the process with the run handler, priority, and flags.
3. Add the process to the `Process_Table` at the correct priority region.
4. Set `p_process__latest`.
5. Return the process pointer (or NULL on failure).

### 4. Per-Frame Scheduling

    // Called once per game tick:
    Quantity__u32 ticks = poll_process_manager(
            &game.process_manager,
            &game);

The scheduler loop:

1. Calls `begin_polling_of__process_table` to reset the cursor to
   priority 0.
2. Calls `poll_next_p_process_from__process_table` repeatedly.
3. For each returned process:
   a. Invokes `m_process_run__handler(p_process, p_game)`.
   b. After the handler returns, checks the process status:
      - **Complete or Failed**: Releases the process via
        `release_process_from__process_manager`. If the process has an
        enqueued process, dequeues it (sets to Idle).
      - **Active**: No action; the process will be polled again next cycle.
      - **Enqueued**: Skipped (should not be returned by the table).
4. Continues until `poll_next_p_process_from__process_table` returns NULL.
5. Returns the number of ticks elapsed.

### 5. Process Release

    release_process_from__process_manager(
            p_game,
            &process_manager,
            p_process);

1. Invokes `m_process_dispose__handler(p_process, p_game)` if set.
2. Removes the process from the `Process_Table` (triggers pointer array
   compaction).
3. Resets the slot to empty (`initialize_process_as__empty_process`):
   UUID = `IDENTIFIER__UNKNOWN__u32`, status = None.

### 6. UUID Lookup

    Process *p_proc = get_p_process_by__uuid(
            &process_manager,
            target_uuid);
        -> Hashed lookup via dehash_identitier_u32_in__contigious_array.
        -> Returns NULL if no process with that UUID exists.

## Integration Points

### Game Action System

Game actions that require multi-step processing are wrapped in processes:

    Game_Action *p_ga = allocate_game_action(...);
    Process *p_proc = run_process(&pm, ga_handler, priority, flags);
    initialize_process_as__game_action_process(p_proc, p_ga);

See `system_overview__game_action_process.md`.

### TCP Communication

TCP multi-packet protocols use game action processes with TCP extensions:

    Process *p_proc = run_process(&pm, tcp_handler, priority, flags);
    initialize_process_as__game_action_process(p_proc, p_ga);
    set_game_action_process_as__tcp_payload_receiver(
        p_game, p_proc, buffer, size);

See `system_overview__tcp_game_action_process.md`.

### File System I/O

File operations are wrapped in filesystem processes:

    Process *p_proc = run_process(&pm, file_handler, priority, flags);
    initialize_process_as__filesystem_process__open_file(
        p_game, p_proc, path, "rb", data, false);

See `system_overview__filesystem_process.md`.

### World Subsystem

The world subsystem uses processes for chunk saving, collision node saving,
and local space node saving:

- `m_process__save_chunk` — saves chunk data cooperatively.
- `m_process__save_collision_node` — saves collision node data cooperatively.
- `m_process__save_local_space_node` — saves local space node data cooperatively.

These are run via `run_process` with appropriate handlers and priorities.

## Capacity Constraints

- The process pool has a fixed size of 512 slots. Allocation fails
  (returns NULL) if all slots are occupied.
- Each active process occupies one pool slot and one pointer in the
  `Process_Table`'s pointer array.
- UUID collisions are resolved by the hashing function's probing strategy.
  In the worst case, probing may fail if the pool is nearly full.

## Error Handling

- `allocate_process_in__process_manager` returns NULL if the pool is full.
- `run_process` / `run_process_with__uuid` return NULL on allocation failure.
- `release_process_from__process_manager` requires a valid `Game*` for
  dispose handler invocation. Debug builds abort on null.
- `get_p_process_by__uuid` returns NULL if no matching process exists.
